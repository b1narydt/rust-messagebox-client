**IMPORTANT distinction from research Pitfall 4:** The `{ message, payment }` wrapper is a server delivery fee mechanism (server-injected). This is DIFFERENT from PeerPay PaymentToken (peer-to-peer, handled by `list_incoming_payments`). The `payment` sub-object here has `tx`, `outputs` (with `InternalizeOutput::WalletPayment` entries), and `description`.

1. Add private helper types at top of http_ops.rs (above the impl block):
```rust
/// Intermediate type for server's wrapped message body format.
/// The server MAY wrap message body as { "message": ..., "payment": ... }
/// where payment contains delivery fee data for internalization.
#[derive(Deserialize)]
struct WrappedMessageBody {
    message: Option<serde_json::Value>,
    payment: Option<ServerPayment>,
}

#[derive(Deserialize)]
struct ServerPayment {
    tx: Option<Vec<u8>>,
    outputs: Option<Vec<serde_json::Value>>,
    description: Option<String>,
}
```

2. Add the public `list_messages` method:
```rust
pub async fn list_messages(
    &self,
    message_box: &str,
    accept_payments: bool,
) -> Result<Vec<ServerPeerMessage>, MessageBoxError> {
```

Logic:
- Call `self.assert_initialized().await?`
- POST to `/listMessages` with `ListMessagesParams { message_box }` (same as list_messages_lite)
- Parse response as `ListMessagesResponse`
- Call `check_status_error`
- For each message in the response:
  a. Try to parse `msg.body` as `WrappedMessageBody`
  b. If parsed AND `payment.tx` is Some AND `accept_payments == true`:
     - Build `InternalizeActionArgs` from the server payment data (tx, description, outputs mapped to `InternalizeOutput::WalletPayment` entries)
     - Call `self.wallet().internalize_action(...)` — log/ignore errors (TS does try/catch and continues)
     - Replace `msg.body` with the `message` sub-field (stringified if Value, or the string if it was a string)
  c. If parsed but no payment or accept_payments=false:
     - Replace `msg.body` with the `message` sub-field
  d. If not parseable as wrapper: leave body as-is (plain message)
  e. Decrypt `msg.body` via `encryption::try_decrypt_message`
- Return the modified messages

NOTE: The server payment internalization is defensive — if parsing fails or internalize_action fails, continue processing. This matches TS behavior where errors are caught and logged.

- `list_messages_plain_body_passthrough`: verify non-wrapped body passes through unchanged
- `list_messages_missing_payment_no_crash`: verify wrapper with `payment: null` works
