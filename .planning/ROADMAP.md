# Roadmap: BSV MessageBox Client (Rust)

## Overview

Six phases translate the TypeScript `@bsv/message-box-client` v1.3.0 into a production Rust crate. Each phase delivers a coherent, independently testable capability. Phase 1 establishes the interoperability foundation — mutex design, encryption format, HTTP core — that all subsequent phases build on. Phases 2 and 3 unlock the primary consumer (`metawatt-edge-rs`) via `CommsLayer` and PeerPay. Phase 4 adds WebSocket live messaging for the 6-node demo. Phase 5 completes overlay host advertisement. Phase 6 performs comprehensive parity verification against the TS client — cross-language integration testing, API surface audit, and smoke testing of every public method.

## Phases

- [x] **Phase 1: HTTP Core + Foundation** - Crate scaffolding, encryption helpers, and basic authenticated HTTP messaging (send, list lite, acknowledge) (completed 2026-03-26)
- [ ] **Phase 2: CommsLayer Adapter + Permissions** - `RemittanceAdapter` implementing `CommsLayer` trait, permission management, notification convenience methods
- [ ] **Phase 3: PeerPay** - Payment token create/send/accept/reject/list and `list_messages` with auto-internalization
- [ ] **Phase 4: WebSocket Live Messaging** - Room-based live messaging with HTTP fallback and live payment listener
- [ ] **Phase 5: Overlay + Device Registration** - Host advertisement, overlay resolution, `init()`, multi-host deduplication, device registration
- [ ] **Phase 6: Parity Verification** - Cross-language integration testing, API surface audit, smoke testing of every public method against TS client

## Phase Details

### Phase 1: HTTP Core + Foundation
**Goal**: A caller can send, list, and acknowledge BRC-31 authenticated messages with correct BRC-78 encryption — and the wire format is byte-compatible with TypeScript clients on the same server
**Depends on**: Nothing (first phase)
**Requirements**: SETUP-01, SETUP-02, SETUP-03, HTTP-01, HTTP-03, HTTP-04, HTTP-05, ENC-01, ENC-02, ENC-03, ENC-04, TEST-01, TEST-02, TEST-03
**Success Criteria** (what must be TRUE):
  1. `cargo build` succeeds with `bsv-rust-sdk` (network feature), `serde`, `tokio`, `base64`, `thiserror` — no direct `hmac`, `reqwest`, or `sha2` dependencies
  2. A message sent from the Rust client can be decrypted and read by the TypeScript client (and vice versa) — confirming protocol `[1, "messagebox"]`, keyID `"1"`, and base64 STANDARD padding all match
  3. `send_message`, `list_messages_lite`, and `acknowledge_message` complete round-trip calls against a running `go-messagebox-server` without auth errors
  4. The HMAC message ID generated by Rust for a given input matches the ID generated by the TypeScript client for the same input
  5. Unit tests for type serialization, encryption round-trip, and HMAC ID generation all pass (`cargo test`)
**Plans:** 2/2 plans complete
Plans:
- [ ] 01-01-PLAN.md — Crate scaffolding, error/wire types, and BRC-78 encryption helpers with round-trip tests
- [ ] 01-02-PLAN.md — MessageBoxClient struct with send_message, list_messages_lite, and acknowledge_message

### Phase 2: CommsLayer Adapter + Permissions
**Goal**: `metawatt-edge-rs` can construct a `RemittanceAdapter`, pass it to `RemittanceManager`, and send/receive/acknowledge messages through the `CommsLayer` trait boundary — with permission management available on the same client
**Depends on**: Phase 1
**Requirements**: PERM-01, PERM-02, PERM-03, PERM-04, NOTF-01, NOTF-02, NOTF-03, NOTF-04, NOTF-05, COMM-01, COMM-02, COMM-03, COMM-04, TEST-04, TEST-05
**Success Criteria** (what must be TRUE):
  1. `RemittanceAdapter<W>` compiles as a valid `CommsLayer` implementation — the compiler accepts it in place of the trait object
  2. Calling `CommsLayer::send_message` via the adapter returns the server-assigned message ID from the HTTP response
  3. Calling `CommsLayer::list_messages` via the adapter returns a `Vec<PeerMessage>` with fields correctly mapped from `ServerMessage`
  4. `set_message_box_permission`, `get_message_box_permission`, `list_message_box_permissions`, and `get_message_box_quote` all return correct data against `go-messagebox-server`
  5. Integration test covering the send → list → ack cycle via `RemittanceAdapter` passes
**Plans:** 1/2 plans executed

### Phase 3: PeerPay
**Goal**: A caller can create, send, accept, and reject BSV payment tokens over MessageBox, and `list_messages` auto-internalizes payments in the retrieved messages
**Depends on**: Phase 2
**Requirements**: HTTP-02, PAY-01, PAY-02, PAY-04, PAY-05, PAY-06, TEST-06
**Success Criteria** (what must be TRUE):
  1. `create_payment_token` produces a payment object whose transaction is accepted by `WalletInterface::internalize_action` on the receiving side
  2. `send_payment` → `list_incoming_payments` → `accept_payment` completes the full payment round-trip against `go-messagebox-server`
  3. `reject_payment` refunds with the correct 1000-satoshi fee deduction (matching TS behavior)
  4. `list_messages` auto-internalizes embedded payment tokens and returns `Vec<PeerMessage>` with decrypted bodies
  5. Integration tests for the payment create → send → accept flow pass
**Plans:** 2 plans
Plans:
- [ ] 03-01-PLAN.md — Payment types, create_payment_token, send_payment, accept_payment, reject_payment, list_incoming_payments
- [ ] 03-02-PLAN.md — list_messages with auto-internalization, PeerPay integration tests

### Phase 4: WebSocket Live Messaging
**Goal**: A caller can subscribe to a WebSocket room, receive live messages with auto-decryption, send live messages with HTTP fallback, and the `CommsLayer` live messaging methods are wired through to WebSocket
**Depends on**: Phase 3
**Requirements**: COMM-05, PAY-03, PAY-07, WS-01, WS-02, WS-03, WS-04, WS-05
**Success Criteria** (what must be TRUE):
  1. `listen_for_live_messages` receives and auto-decrypts a message sent by another client in the same room
  2. `send_live_message` delivers via WebSocket when connection is healthy; falls back to HTTP `send_message` when WebSocket is unavailable or times out after 10 seconds
  3. `join_room` and `leave_room` correctly subscribe and unsubscribe — a message sent to a room after `leave_room` is not delivered to the unsubscribed callback
  4. `disconnect_web_socket` closes the connection cleanly with no leaked tasks or pending ack channels
  5. `CommsLayer` live messaging methods (previously default impls) are wired to WebSocket operations

### Phase 5: Overlay + Device Registration
**Goal**: The client can auto-initialize by querying and advertising its MessageBox host via the overlay network, resolve recipient hosts via overlay lookup, deduplicate messages across multiple hosts, and register devices for push notifications
**Depends on**: Phase 4
**Requirements**: HOST-01, HOST-02, HOST-03, HOST-04, HOST-05, HOST-06, HOST-07, DEV-01, DEV-02
**Success Criteria** (what must be TRUE):
  1. `init()` called concurrently by multiple tasks executes the overlay advertisement exactly once (no duplicate `anoint_host` transactions)
  2. `resolve_host_for_recipient` returns a host URL that can be used to send a message the recipient retrieves successfully
  3. `anoint_host` broadcasts a valid PushDrop transaction the overlay accepts; `revoke_host_advertisement` spends it cleanly
  4. `list_messages` across two advertised hosts returns a deduplicated message set — a message present on both hosts appears exactly once
  5. `register_device` and `list_registered_devices` complete against `go-messagebox-server` without errors
**Plans**: TBD

### Phase 6: Parity Verification
**Goal**: Every public method in the Rust crate behaves identically to its TypeScript counterpart — confirmed by cross-language integration tests, API surface audit, and comprehensive smoke testing
**Depends on**: Phase 5
**Requirements**: PARITY-01, PARITY-02, PARITY-03, PARITY-04, PARITY-05
**Success Criteria** (what must be TRUE):
  1. API surface audit confirms all 28 TS `MessageBoxClient` methods + 7 `PeerPayClient` methods have Rust equivalents with matching signatures
  2. Cross-language integration test: Rust client sends encrypted message → TS client decrypts successfully (and vice versa) on the same `go-messagebox-server`
  3. Cross-language payment test: Rust `create_payment_token` → TS `acceptPayment` succeeds (and vice versa)
  4. Smoke test exercises every public method against a live server — all return expected results with no panics or unhandled errors
  5. Wire format comparison: JSON request/response bodies from Rust client match TS client byte-for-byte for identical operations (captured via server logs or proxy)
