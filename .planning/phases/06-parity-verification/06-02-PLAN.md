**Step 1 — Add acknowledge_notification to src/peer_pay.rs:**

```rust
pub async fn acknowledge_notification(
    &self,
    message: &bsv::remittance::types::PeerMessage,
) -> Result<bool, MessageBoxError>
```

Implementation (composition pattern — see RESEARCH.md Pattern 4):
1. Try to parse `message.body` as a wrapped envelope `{"message": ..., "payment": ...}`
2. If payment is present and has a valid `tx` field:
   a. Extract the payment data into an `IncomingPayment` struct
   b. Call `self.accept_payment(&incoming)` which both internalizes AND acknowledges
   c. Return `Ok(true)` (payment was internalized)
3. If no payment or parsing fails:
   a. Call `self.acknowledge_message(vec![message.message_id.clone()], None)`
   b. Return `Ok(false)`

Use the existing `WrappedMessageBody` struct from http_ops.rs for parsing (or add `#[derive(Deserialize)]` if it's not already public). The wrapped body format is `{ message: String, payment: Option<ServerPaymentOutput> }` — same struct used in `list_messages` payment internalization.

Note on input type: Use `bsv::remittance::types::PeerMessage` (from the bsv-sdk). This is what `list_messages` returns and what callers will have. If this type is not accessible, use a minimal struct with `message_id: String` and `body: String` fields. Check the actual import path — it may be `bsv::wallet::types::PeerMessage` or similar.

**Step 2 — Add send_message_to_recipients to src/http_ops.rs:**

```rust
pub async fn send_message_to_recipients(
    &self,
    params: &crate::types::SendListParams,
    override_host: Option<&str>,
) -> Result<crate::types::SendListResult, MessageBoxError>
```

Implementation (see RESEARCH.md Pattern 6):
1. Call `get_message_box_quote_multi` (from Step 3) to get per-recipient quotes
2. Separate recipients into: blocked (from quote), payable, always_allow
3. If there are payable recipients, call `create_message_payment_batch` to create a single payment transaction
4. Loop over non-blocked recipients, calling `send_message_to_host` for each with the shared batch payment
5. Collect results into `SendListResult { sent, blocked, failed, totals }`

Key insight from TS: this is NOT a single multi-recipient HTTP call. It loops individual `sendMessage` calls. The batch payment is a single transaction with multiple outputs.

The `skip_encryption` from `SendListParams` flows through to each `send_message_to_host` call.
